# MusicKeyboard Sampler Research Notes

정리 날짜: 2026-04-15 ~ 2026-04-16

## 목표 정리

만들고 싶은 기기 방향은 아래에 가깝다.

- 단순한 임베디드 악기
- 작은 스피커 또는 3.5mm 출력 사용
- 마이크로 짧게 녹음
- 재생 속도 변경으로 pitch 변화
- 고음질 악기보다는 로파이 샘플러/토이 악기 성격
- 복잡한 time-stretch보다는 속도 변화 기반 playback

이 목표라면 `RP2350 단독`으로도 충분히 가능하다.

## MCU 쪽 메모

### 왜 RP2040 / RP2350 계열이 많이 쓰이는가

- USB device 기능이 칩에 내장되어 있어서 PC에 바로 키보드/HID/오디오 장치처럼 붙이기 쉬움
- QMK 같은 키보드 생태계가 이미 두꺼움
- GPIO가 넉넉하고 핀 멀티플렉싱이 유연함
- UF2 드래그앤드롭 플래싱이 쉬움
- 가격 대비 성능이 좋음

### RP2350를 써도 되는가

가능하다. 오히려 여유가 더 있다.

- RP2040보다 RAM/성능 여유가 큼
- 샘플 재생, UI, 시퀀서, MIDI, I2S 입출력 같이 돌리기 편함
- 로파이 샘플러 목적에는 충분 이상

결론:

- 일반적인 로파이 샘플러 / 미니 악기 목적이라면 `RP2350`으로 충분
- LMN-3 같은 큰 구조는 과함

## 오디오 입력 쪽

### 아날로그 마이크 vs I2S MEMS 마이크

현재 목표에는 `I2S MEMS 마이크`가 더 잘 맞는다.

이유:

- 아날로그 프리앰프/ADC 튜닝 부담이 줄어듦
- 노이즈 문제를 덜 겪음
- RP2350와 연결하기 좋음
- 작은 기기에 넣기 좋음

### 후보 1: ICS43434

장점:

- INMP441보다 스펙이 조금 더 좋음
- 전류 소모가 적은 편
- 감도 공차가 더 좋음
- 로파이 샘플러 용도로 충분

단점:

- 현재는 구형/수명주기 측면에서 애매함
- 모듈 수급은 판매처에 따라 다를 수 있음

### 후보 2: INMP441

장점:

- 메이커 시장에서 모듈이 흔함
- 예제, 배선 사례가 많음
- 알리 같은 곳에서 찾기 쉬움

단점:

- ICS43434보다 칩 스펙은 약간 떨어짐

### 후보 3: T5848

장점:

- 현행 제품
- 칩 스펙이 좋음

단점:

- 취미용 브레이크아웃/알리 모듈이 드묾
- 초반 프로토타입 용도로는 접근성이 낮음

### 실제 추천

프로토타입:

- `INMP441 모듈` 또는 `ICS43434 모듈`

판단:

- 빠르게 구해서 테스트할 거면 `INMP441`
- 둘 다 쉽게 구할 수 있다면 칩만 보면 `ICS43434` 쪽이 조금 더 나음

### 펌웨어 차이

`INMP441`과 `ICS43434`는 둘 다 I2S 디지털 마이크라서 펌웨어 구조는 거의 비슷하다.

보통 달라지는 건:

- 핀 설정
- L/R 선택
- 샘플레이트
- 24-bit 데이터를 내부에서 16-bit/8-bit로 줄이는 처리

## MEMS 마이크 포트 방향 메모

많은 MEMS 마이크는 `bottom-port` 구조다.

의미:

- 마이크 입구가 칩 윗면이 아니라 아랫면에 있음
- PCB에 acoustic hole이 필요함
- 케이스에도 해당 위치에 맞는 구멍이 필요함

즉:

- `케이스 구멍 -> PCB 홀 -> 마이크 하단 포트`

주의:

- 포트 주변 막힘이 있으면 수음이 나빠짐
- 플럭스 잔여물, 테이프, 폼 등으로 막히면 안 좋음
- 스피커와 너무 가까우면 피드백/공진 가능

## 오디오 출력 쪽

### 가장 중요한 구분

출력은 크게 두 종류다.

- `스피커 직접 구동용`
- `라인아웃 / 이어폰 / 외부앰프 입력용`

이 둘은 다르게 설계해야 한다.

### MAX98357A

특징:

- I2S 입력을 받아 작은 스피커를 직접 울리는 `모노 Class-D 앰프`
- 아주 작고 흔하고 싸며 예제가 많음

장점:

- 구현이 쉬움
- RP2350와 연결하기 쉬움
- 작은 스피커 프로토타입에 적합

중요:

- `이어폰 잭 직결용 아님`
- 출력은 일반적인 L/R/GND 라인아웃이 아니라 스피커용 BTL 출력
- 보드에 보이는 `L`은 보통 I2S의 LRCLK 쪽 의미이며, 이어폰 Left 출력이 아님

결론:

- `작은 스피커 전용 프로토타입`에는 아주 좋음
- `라인아웃/이어폰` 용도로는 부적합

### TLV320DAC3100

특징:

- 출력용 DAC
- 헤드폰/라인/스피커 쪽 확장이 가능함
- 보드에 `MIC`나 `BIAS`가 보일 수 있지만, 이 칩은 본질적으로 `출력용 DAC`

주의:

- `녹음용 ADC 칩 아님`
- 보드의 MIC/BIAS는 헤드셋 마이크 바이어스나 출력 믹서 쪽 개념
- 마이크를 MCU로 디지털화해서 넘겨주는 입력 codec으로 보긴 어려움

의미:

- 출력 쪽 최종형 후보로 좋음

### 그 외 출력 후보

#### PCM5102A

- line out 용으로 단순하고 무난
- 스피커 직접 구동용은 아님
- 이어폰도 별도 앰프 없이 직결용으로 보기는 애매함

#### WM8960 모듈

- codec + 스피커 앰프 + 헤드폰 앰프 + 마이크 입력까지 포함된 쪽
- 기능은 좋지만 부피가 커질 수 있음

#### SGTL5000

- codec + 헤드폰/라인 출력 쪽은 좋음
- 스피커 파워앰프는 따로 필요

## 라인아웃 관련 정리

라인아웃을 제대로 보내려면 보통 `DAC`가 필요하다.

이유:

- 라인아웃은 `GND 기준의 작은 아날로그 오디오 신호`
- 스피커 앰프 출력과 다름

즉:

- `스피커 직결` -> MAX98357A 같은 I2S 앰프
- `라인아웃 / 이어폰 / 외부앰프 입력` -> DAC 또는 codec

- 입력: `INMP441` 또는 `ICS43434`

장점:

단점:

- 설정과 회로가 조금 더 복잡

## 스피커 선택 메모

### 4Ω vs 8Ω

- `4Ω`은 더 큰 출력이 나올 수 있지만 앰프 부담이 큼
- `8Ω`은 출력은 조금 낮을 수 있지만 다루기 쉽고 안정적

### 3W, 2W의 의미

- 스피커가 견딜 수 있는 최대 출력의 개념
- 실제로 항상 그 와트만큼 쓰는 건 아님

### 현재 추천

작은 샘플러/장난감 악기라면 `8Ω 3W`도 괜찮다.

이유:

- 앰프 부담이 비교적 덜함
- 여유 있게 버틸 수 있음
- 실제 음량은 볼륨 조절 가능

단, 실제 크기와 체감 음량은 아래 영향을 더 많이 받음:

- 스피커 효율
- 스피커 지름
- 인클로저 구조
- 전원 상태

## 펌웨어 구현 방향

### 우선순위

1. 내장 샘플 재생
2. playback speed 변경으로 pitch 변화
3. 마이크 녹음
4. 녹음한 샘플 저장

`time-stretch`는 하지 말고 `속도 변화 = pitch 변화`로 가는 것이 좋다.

이유:

- 구현이 단순함
- RP2350 단독 구조에 잘 맞음
- 오히려 로파이 악기 감성에 더 잘 맞음

## UI/입력 하드웨어 결정 — D200 + RP2350 직결

### 왜 D200인가

샘플러에 버튼이 많이 필요하다 (슬롯 선택, FX 토글, 파라미터 조절 등). D200은 5인치 720×1280 LCD + 13버튼을 하나의 하우징으로 제공한다.

처음엔 `D200 + ESP32-S3-ETH` 조합을 검토했으나, 샘플러는 네트워크가 불필요한 스탠드얼론 기기라서 ESP32는 불필요한 중간 단계가 된다. `D200 + RP2350` 직결이 더 단순하고 오디오 DSP에도 더 유리하다.

### 레이턴시 검증

UART 경로 레이턴시:
```
버튼 누름 → Linux input event (1~2ms) → ulanzi-control 데몬 (~1ms) → UART 0.16ms → RP2350
                                                                        합계: 2~5ms
```

- UART 1.5Mbps로 30바이트 JSON 전송은 0.16ms — 무시 가능
- 병목은 Linux 스케줄링 지터
- 로파이 토이 악기 용도에서 2~5ms는 충분히 허용 가능

### UART 프로토콜 구현

양방향 NDJSON, `\n` 구분자. 프로토콜은 고정되어 있으므로 RP2350에서 파서만 구현하면 된다.

**RX (D200 → RP2350)**: 버튼 이벤트 수신
```json
{"type":"button_press","id":0}
{"type":"button_release","id":0}
```

**TX (RP2350 → D200)**: 화면 업데이트 송신
```json
{"type":"set_button","id":0,"text":"120BPM","icon":5}
{"type":"set_button","id":1,"value":0.75}
```

프로토콜 레퍼런스: [commands.go](https://github.com/mapero/esphome-ulanzi-d200/blob/main/daemon/pkg/transport/commands.go)

RP2350 구현 순서:
1. UART 1.5Mbps 설정
2. `\n` 기준 라인 버퍼링
3. JSON 파싱 → 버튼 ID 추출 → 샘플러 트리거
4. 샘플러 상태 변경 시 → JSON 포맷 → UART 송신

## 한 줄 결론

지금 목표에 가장 잘 맞는 방향은 아래다.

- `RP2350` (오디오 DSP + D200 UART 통신)
- `D200` (5인치 화면 + 13버튼 UI)
- `I2S MEMS 마이크 모듈`
- `속도 변화 기반 로파이 샘플 재생`
- `초기 출력은 MAX98357A + 8Ω 스피커`
- 이어폰/라인아웃은 나중에 DAC/codec 단계에서 확장
